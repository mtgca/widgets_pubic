Sinusoidal Positional Encoding: Computed, Not Learned

In the original Transformer paper the position table is not an nn.Embedding that training must fill in. Every row is computed by a fixed sin/cos formula and registered as a buffer — zero trainable parameters, and a valid vector exists for any position, even ones the model never saw. This widget is the companion of Trainable Positional Embeddings: same corpus, same model, same training recipe — only the position table changes.

The whole idea

Swap the learned table for a formula. Everything else in the model stays identical.

# companion widget (trainable):
self.position_embedding = nn.Embedding(n_positions, n_embed)

A parameter matrix — starts random, learned by backprop, fixed number of rows.

# this widget (sinusoidal, frozen):
self.register_buffer('position_embedding', get_sinusoidal_position_embeddings(num_positions, n_embed))

A buffer — computed once from sin/cos, never updated, extendable to any length.

The big picture · what still learns, what doesn’t

Backpropagation now skips the position table

The forward pass is the same as in the trainable version — word lookup + position lookup → sum → linear → loss. The difference is the backward pass: a buffer is not a parameter, so the gradient simply has nothing to update there.

Same architecture as the trainable-embedding widget — but only the blue and white boxes contain parameters. The green box is pure arithmetic.

Step 1 · The formula and the table it generates

Every entry is a sine or cosine — nothing is stored, everything is computable

Even columns get a sine, odd columns get a cosine, and each pair (2i, 2i+1) uses its own frequency ωi — this is exactly what the classroom get_sinusoidal_position_embeddings() computes (div_term holds the ωi values).

PE(p,2i) = sin(p·ωi) , PE(p,2i+1) = cos(p·ωi) , ωi= 1100002i/d

Read it like a heatmap: blue = negative, red = positive, every value in [−1, 1]. The left column pairs oscillate fast down the table (high ωi), the right pairs barely move (tiny ωi) — together they give every position a unique fingerprint. Click any row to select it. Note the values never grow with p: position 47 is as well-behaved as position 0.

A learned table stops at its last row. This formula happily returns a vector for position 1 000 or 1 000 000 — no retraining, no resizing.

Step 2 · Intuition: clock hands at geometric speeds

Each sin/cos pair is one hand of a clock — slide the position and watch them turn

A pair (sin(p·ω), cos(p·ω)) is a point on a unit circle: a hand at angle p·ω. With d=8 there are four hands, each turning 10× slower than the previous one — like seconds, minutes, hours. Fast hands pinpoint the fine position; slow hands disambiguate the coarse one.

p = 5

Why several frequencies? One fast hand alone repeats: positions 0 and ~6 would look almost the same (sin has period 2π). One slow hand alone barely distinguishes neighbors. The combination is unique over an enormous range — the same trick as a clock face, or as digits in a number (units, tens, hundreds…).

Why the constant 10000? It sets the speed of the slowest hand, i.e. the longest “period” of the encoding. A large base means even sequences of thousands of tokens never wrap around to a repeated fingerprint.

Step 3 · Built-in sense of distance

How similar are two positions? It only depends on how far apart they are

Dot the fingerprints of two positions and the sines and cosines collapse (sin⁢a·sin⁢b+cos⁢a·cos⁢b=cos(a−b)), leaving a value that depends only on the offset k — not on where the pair sits in the sentence.

PE(p) · PE(p+k) = ∑i=0d/2−1 cos(k·ωi)
k = 3

Check it numerically — three different starting positions, same offset, identical dot product:

Why this matters — shifting is a rotation. Advancing any position by k rotates each hand by the fixed angle k·ωi, no matter where the hand started:

( sin((p+k)ω) cos((p+k)ω) ) = ( cos(kω) sin(kω) −sin(kω) cos(kω) ) ( sin(pω) cos(pω) )

So PE(p+k) is a linear function of PE(p). The paper’s authors chose sinusoids precisely for this: it should let attention easily learn to look “3 tokens back” as one fixed linear operation, valid everywhere in the sequence. A trainable table has no such structure — it must discover any notion of “distance” from data alone.

Step 4 · The classroom model, trained in your browser

Same recipe as the trainable widget — but zero positional parameters

Full-batch Adam, lr = 0.1, 300 epochs of cross-entropy on the same 34 (word, position, next-word) triplets from the 14-sentence corpus — matching the “sinusoidal positional embeddings” section of the classroom notebook. The position rows are exactly the sin/cos values from Step 1; they receive no gradient.

What is actually trained:

word_embedding · 11×8 = 88 params · trained linear · 8×11 + 11 = 99 params · trained position_embedding · 0 params · frozen buffer

In the trainable companion the position table added 4×8 = 32 learned parameters. Here it adds none — and the model still reaches a comparable loss. The paper reports the same finding at full scale: learned and sinusoidal encodings gave nearly identical results.

The word table and linear layer re-learn from new random values — the green position rows are bit-for-bit identical every run. Deterministic by construction.

Step 5 · The killer advantage: positions that were never trained

Feed the model positions 4 and 5 — a learned table couldn’t even try

The corpus only ever puts input words at positions 0–3, so the trainable companion built a 4-row table: asking it for position 5 is an index out of range crash. The sinusoidal formula just computes row 5. Pick a word and compare its predictions across positions — including two the model has never seen.

Honest caveat: at positions 4–5 the input vector is perfectly valid, but this toy model never trained there, so treat those predictions as “the model accepts it gracefully”, not “the model is right”. In real Transformers this is exactly why the paper chose sinusoids: the model may extrapolate to sequence lengths longer than the ones encountered during training.

# trainable table (nn.Embedding(4, 8)):
model.position_embedding(torch.tensor(5))
IndexError: index out of range in self
# sinusoidal buffer — any row on demand:
get_sinusoidal_position_embeddings(6, 8)[5]
tensor([-0.9589, 0.2837, 0.4794, 0.8776, ...])
Step 6 · Trainable vs. sinusoidal — the honest scorecard

Two valid designs, one trade-off

Trainable table (companion widget) Sinusoidal encoding (this widget)
Extra learned parameters num_positions × n_embed (here 4×8 = 32; in GPT-2, 1024×768 ≈ 786k) Zero — the table is pure arithmetic
Position beyond training length Impossible — the row doesn’t exist (IndexError) Computed like any other — may extrapolate to longer sequences
Values Whatever SGD finds — different every run, shaped by the task Deterministic — identical for every model, run, and machine
Relative offsets Must be discovered from data Built in — shifting by k is a fixed rotation of every sin/cos pair
Can it adapt to the task? Yes — the rows are free parameters No — fixed forever (the price of the other advantages)
Accuracy in the paper (Table 3, row E) Nearly identical — the authors picked sinusoids for the extrapolation hope, not for accuracy

Epilogue for the curious: BERT and GPT-2 went with learned tables anyway; most modern LLMs moved on to rotary embeddings (RoPE) — which take the “shift = rotation” idea of Step 3 and apply it directly inside attention.

The classroom code, annotated

def get_sinusoidal_position_embeddings(num_positions, n_embed):
    pe = torch.zeros(num_positions, n_embed)
    position = torch.arange(0, num_positions, dtype=torch.float).unsqueeze(1)
    div_term = torch.exp(torch.arange(0, n_embed, 2).float() * -(math.log(10000.0) / n_embed))  # the frequencies ω i
    pe[:, 0::2] = torch.sin(position * div_term)   # even columns  = sin(p·ω i)
    pe[:, 1::2] = torch.cos(position * div_term)   # odd columns   = cos(p·ω i)
    return pe                                       # no nn.Parameter anywhere — just a tensor

class SinusoidalPositionalEmbeddingLM(nn.Module):
    def __init__(self, vocab_size, n_embed, num_positions):
        super().__init__()
        self.word_embedding = nn.Embedding(vocab_size, n_embed)          # trainable (blue box)
        self.register_buffer('position_embedding',
                             get_sinusoidal_position_embeddings(num_positions, n_embed))  # frozen (green box)
        self.linear = nn.Linear(n_embed, vocab_size)                      # trainable

    def forward(self, word_idx, pos_idx):
        word_emb = self.word_embedding(word_idx)     # lookup in a learned table
        pos_emb  = self.position_embedding[pos_idx]  # lookup in a computed table
        emb = word_emb + pos_emb                     # same element-wise sum as before
        return self.linear(emb)

register_buffer is the key line: the tensor moves with the model (.to(device), state_dict) but is invisible to the optimizer — model.parameters() never yields it, so opt.step() can never touch it.

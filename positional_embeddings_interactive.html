Trainable Positional Embeddings: Same Table, Different Index

A positional embedding is not a new mechanism. It is a second nn.Embedding — the very same lookup-table object used for words. The only difference is where the row index comes from: the word table is indexed by which token it is, the position table by where the token sits in the sentence. The model below was trained live in your browser with the same PyTorch recipe from the “Trainable Positional Embeddings” section of this notebook.

The whole idea

Two copies of the same layer. One is indexed by token identity, the other by position.

self.word_embedding = nn.Embedding(vocab_size, n_embed)

Row index = stoi[word]  ·  “which word is this?”

self.position_embedding = nn.Embedding(n_positions, n_embed)

Row index = position p  ·  “where in the sentence is it?”

The big picture · model architecture

One training triplet flowing through the whole model

Two parallel lookups — word branch and position branch — meet at an element-wise sum, then a linear layer scores every vocabulary word. The loss compares those scores against the word that actually came next, and the backpropagated gradient updates both tables the same way.

Schematic, drawn for one training triplet: input word “pizza” at position 2, target “and” (from “we like pizza and pasta”). The steps below walk through these same layers with the real, learned numbers.

Step 2 · Two lookups, one mechanism

The same operation runs twice — only the index differs

Both tables below are learned parameter matrices with n_embed = 8 columns. A lookup just returns one row. Nothing else happens.

Word embedding table

word_embedding.weight — shape (vocab_size = 11, n_embed = 8)

Position embedding table

position_embedding.weight — shape (n_positions = 4, n_embed = 8)

Notice the position table has fewer rows (4) only because this toy corpus never feeds the model a word beyond position 3 — its “vocabulary” is the set of positions {0, 1, 2, 3}, just as the word table’s vocabulary is the set of 11 words. Both tables start random and both are updated by the same backpropagation step.

Step 3 · Add the two rows

The input to the rest of the network is the element-wise sum

emb= Eword [stoi(w)] + Epos [p] , logits=W·emb+b , y^= softmax(logits)

Technical note — why add instead of concatenate? We add the two vectors so the result keeps the same size, d. If we glued them together instead, the vector would double in length, and every layer after it would need to grow too. The embedding tables themselves cost the same either way — the savings is in the rest of the network.

Doesn’t mixing two signals into one vector scramble them? In principle it could, but with many dimensions to work with, the model has plenty of room to store what the word means and where it sits along different directions of the vector — and since both tables are learned, training figures this out on its own. So the layers that follow can still read each signal separately.

Back to the code

The forward pass you already know

def forward(self, word_idx, pos_idx):
    word_emb = self.word_embedding(word_idx)   # row lookup, index = which word   (blue table)
    pos_emb  = self.position_embedding(pos_idx) # row lookup, index = which slot   (green table)
    emb = word_emb + pos_emb                    # element-wise sum                 (purple row)
    logits = self.linear(emb)
    return logits

Step 1Projecting Queries, Keys, and Values

Each token is transformed into three distinct functional roles: Query, Key, and Value.

The input x∈ℝn×d holds one embedding row per token (semantic + positional information). We multiply it by three learned weight matrices WQ,WK,WV to project every token into a new feature space. This lets the model learn what to “look for” (Query) and what to “show” (Key) independently.

Hover any cell of the result to see the dot product that produces it — one row of x (a token) against one column of the weight matrix.

Sequence length n

The number of tokens in the current context window. Here n=4.

Model dimension d (embedding dim)

The width of the hidden representation (e.g. 512 in real models). Here d=4.

Projection dimension dk (head dim)

The width of the resulting Q,K,V vectors — usually smaller than d (Step 6: dk=d/h). Here dk=3.

Technical note

In general WV∈ℝd×dv while WQ,WK∈ℝd×dk; although dk and dv can differ, in practice it is common to set dk=dv (we do here).

Step 2Quantifying Affinities: the Scaled Dot-Product

How similar is each Key to each Query?

The dot product Q·KT measures the affinity between a Query (what a token is looking for) and every Key (what other tokens are offering). Aligned vectors give a large positive product (high affinity); near-orthogonal or opposite vectors give a small or negative one (low affinity).

Hover any cell of the affinity matrix: it highlights the Query row and Key column it comes from — and its mirror cell across the diagonal, to show the matrix is not symmetric.

Why scale by 1/dk? Variance control

As dk grows, dot products grow too. If Q and K entries are independent with mean 0 and variance 1, their dot product has variance dk. Dividing by dk pushes the variance back to 1 — numerical stability for the softmax that comes next.

The result is n×n

One affinity score for every (query token, key token) pair — QKT∈ℝn×n. Row i = token i asking; column j = token j answering.

Technical note: dot product vs. cosine similarity

Q·KT is related to cosine similarity, but we omit the magnitude normalization (dividing by ∥Q∥∥K∥). The scaled dot-product is computationally cheaper and, with the 1/dk scaling, numerically stable for deep networks.

Step 3From Affinities to Attention Weights: Where to Attend

Softmax turns raw, unbounded scores into a probability distribution per row — the “filter”.

αi,j= softmax (QKTdk) ∈ℝn×n ∑jαi,j=1

Softmax makes the highest affinities stand out and pushes low scores toward zero. Each row is now a set of attention weights: a weight of 0.64 means the token is “paying 64% attention” to that word. Click a row of the matrix to read it as a distribution.

Every row sums to 1.00 — each token has an attention “budget” of exactly 1 to spend.

It’s NOT symmetric! Non-constant diagonal The attention matrix is a directed, asymmetric graph — not a correlation table.

Intuition: “Guatemala” vs. “bank”

A highly specific token (Guatemala) already carries rich, unambiguous meaning — it tends to keep a high self-attention score. An ambiguous token (bank) is useless alone: it must look outward (to river or money) to resolve its identity — low diagonal.

Step 4Aggregating Value through Attention Weights

Each output row is a weighted sum of all Value vectors — a context-aware representation.

Pick a token: its new representation zi blends every Value row, each contributing in proportion to its attention weight αi,j. Tokens with near-zero weights are effectively filtered out.

The “weighted average” effect

High-weight tokens contribute more of their Value to the result; irrelevant tokens (near-zero weights) barely register.

Final output Z=A(Q,K,V)

Same number of rows as the input (n), but now every token “knows” its relationship to the rest of the sentence — one context-aware feature vector per token.

Step 5Summary: One Self-Attention Head

Attention is the foundational building block of the Transformer architecture.

Z= A(Q,K,V) =softmax (QKTdk) V ∈ℝn×dk
  1. Encode position and semantic information into embeddings (see the positional-embeddings widget).
  2. Project tokens into three specialized roles: Query, Key, and Value.
  3. Compute attention weights by matching Queries to Keys (scaled dot-product + softmax).
  4. Extract context-aware features by applying those weights to the Values.

Step 6Multi-Head Self-Attention

Learning multiple relationships simultaneously.

The limitation of a single head: one attention head can only focus on one type of relationship at a time (say, subject–verb agreement). The multi-head solution: run h heads in parallel, each attending to information from a different representation subspace. Each head gets a slice of the model width: dk=d/h.

Number of heads h (with d=6):

Feature diversity

Head 1 might focus on syntax (grammar rules); Head 2 on semantics (meaning, synonyms); Head 3 on long-range dependencies (a word mentioned paragraphs ago).

Why dk=d/h?

Concatenating h heads of width d/h restores width d: the output shape matches the input shape — strictly necessary for residual connections (which help with vanishing gradients) — and keeps the model fast and cheap (no parameter blow-up as heads are added).

The “ensemble” intuition

Like an ensemble in classical ML: instead of one “expert”, 8 or 16 parallel heads each look for different clues (syntax, semantics, long-range links) — and all of it runs in parallel.

The role of WO

If the heads are individual experts examining different parts of the problem, WO∈ℝd×d is the project manager: it gathers their raw reports and synthesizes them into one unified representation the rest of the network can act on.

Why Not Just Flatten the Sequence into One Dense Layer?

Suppose we want a layer that lets every token in a sequence mix information with every other token. The simplest idea: flatten the whole sequence into one giant vector and feed it through a single fully-connected (Dense) layer. Slide N (sequence length) and d (features per token) below and watch what happens to the layer's size.

The naive idea: flatten, then fully-connect

Each token is a vector of d numbers. A sequence of N tokens, flattened end-to-end, is one vector of length N·d. A Dense layer connects every input number to every output number — so its weight matrix must be (N·d) ×(N·d).

The parameter count explodes

Doubling the sequence length doesn't double the cost — it roughly quadruples it. The same is true for doubling d. That's what it means for a cost to scale as N2·d2.

🧩 Flattening throws away structure

Treating the sequence as one long vector forgets that it's made of N separate tokens — position and token boundaries only survive because of where a number happens to sit in the vector.

A layer built this way has no idea "these d numbers are token 3" — it just sees N·d numbers.

📈 Two multiplicative culprits

The cost has two independent sources of growth: more tokens (N) and richer token representations (d). Because the weight matrix is squared in the combined size, each one contributes its own squared factor.

That's why the exponent lands on both: N2 and d2, not just one of them.

🚧 This doesn't scale to real sequences

Real sequences run into the hundreds or thousands of tokens, with hidden sizes in the hundreds. Plug those into the counter above and the parameter count blows past anything trainable.

This is the problem that motivates looking for a fundamentally different kind of layer.
